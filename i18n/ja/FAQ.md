# よくある質問 (FAQ)

## SySL について

### なぜAI生成コードには特別なライセンスが必要なのですか？

従来のオープンソースライセンス（MIT、GPL、Apache-2.0）は著作権法の上に構築されています。それらは、人間の著作者がコードの著作権を保有することを前提としています。しかし、米国（Thaler v. Perlmutter, 2025-2026）およびEUの裁判所は、純粋にAIによって生成された著作物は著作権で保護され得ないと判断しています——人間の著作者が存在しないため、著作権も存在しないのです。

AI生成コードに著作権がない場合、著作権に依拠する従来のライセンスは法的効力を持たない可能性があります。SySL は、著作権ライセンスとしても（著作権が存在する場合）、独立した契約としても（存在しない場合）機能することで、この問題を解決します。

### SySL は「オープンソース」ですか？

SySL は、自由ソフトウェアの定義（実行、研究、改変、共有の自由）およびオープンソースの定義の大部分を満たしますが、一つの例外があります：SySL はAI生成の開示を要求します。この開示要件は使用上の条件であり、OSDの一部の解釈では制限とみなされる可能性があります。

SySL は**新しいカテゴリ**——「合成ソース (Synthetic Source)」——を定義します。これはオープンソースに隣接し互換性を持ちながら、従来のオープンソースが決して設計対象としなかったシナリオに対処するものです。

### SySL はコピーレフトライセンスですか？

はい、しかし新しい種類です。従来のコピーレフト（GPL）は、派生物が同一のライセンスの下で配布されることを要求します。SySL の「合成的コピーレフト」は、派生物が **AI開示の連鎖を保持すること** のみを要求します——派生物が SySL の下で配布されることではありません。

したがって、SySL でライセンスされたコードを取得し、改変し、その結果を MIT または Apache-2.0 の下で配布することができます。ただし、**AI開示をそのまま保持する限りにおいて**です。これは意図的なものです：SySL は、ライセンスの非互換性を生み出すことなく、透明性の規範を広めることを目指しています。

## SySL の利用

### SySL でライセンスされたコードをプロプライエタリ製品で使用できますか？

はい。ただし、開示要件（第2条）および相互開示要件（第8条）を満たすことを条件とします。これは以下のことを意味します：

- AI開示通知をドキュメンテーションに保持すること。
- ソースコードを配布する場合、ファイルレベルでAI生成開示を維持すること。
- バイナリのみを配布する場合、（開示を含む）対応するソースコード、またはそうする旨の書面による提供を提供すること。

### SySL コードを Apache-2.0 コードと組み合わせることができますか？

はい。SySL は Apache-2.0 と明示的に互換性があります。組み合わせた著作物は、SySL の開示要件が独立して満たされることを条件として、Apache-2.0 の下で配布することができます。

### SySL コードを GPL コードと組み合わせることができますか？

GPL との互換性は正式に分析されていません。両方のライセンスはコピーレフト類似の義務を持っています（GPLはソース開示、SySL はAI開示）。SySL コードを GPL コードと組み合わせる必要がある場合は、法律顧問に相談するか、互換性分析のためにIssueを作成してください。

### SySL を BUSL と併用できますか？

はい。SySL は階層的ライセンス供与のために設計されています。ライセンサーは、ソフトウェアを SySL と BUSL-1.1 の両方の下で同時に提供することができます（デュアルライセンス）。これら二つのライセンスは異なる目的を果たします：

- **BUSL-1.1**：開発期間中の商業的利用を保護し、その後指定された変更ライセンスに移行します。
- **SySL**：AI生成が開示されなければならないという透明性の規範を確立します。

利用者は、いずれのライセンスに従うかを選択できます。正式な階層的ライセンス供与の規定については、ライセンス第11.3項を参照してください。

**重要**：AI生成ソフトウェアに対して BUSL を SySL と併用する場合、BUSL の変更ライセンスは SySL に設定すべきです（Apache-2.0 や MIT ではなく）。BUSL が開示義務のないライセンス（Apache-2.0 など）に移行した場合、変更日以降のコードの受領者がAI開示を除去できてしまい、SySL の目的が損なわれます。変更ライセンスを SySL に設定することで、BUSL の競業避止期間後も透明性の規範が存続します。

### ソフトウェアを内部でのみ使用する場合、AI使用を開示する必要がありますか？

第2条の開示要件は、ソフトウェアを**配布する**場合に適用されます。単一の組織内での内部使用は、SySL の下での配布を構成しません。ただし、ソフトウェアを第三者がアクセス可能なサービスとして展開する場合、第8.2項が適用される可能性があります。

### 何が「AI生成」とみなされますか？

大規模言語モデル（LLM）、コード生成モデル、および類似の機械学習システムを含む、AIモデルまたはシステムによって生成されたコード、ドキュメンテーション、設定、またはその他のあらゆる素材です。これには、部分的に生成された後、人間によって編集されたコードも含まれます——ファイルレベルの開示では、おおよそのAI生成割合を記載する必要があります。

## 法的な質問

### SySL は法的に執行可能ですか？

SySL は、二つの独立した法理論の下で執行可能となるように設計されています：

1. **著作権法**：AI生成コードが著作権の対象となると判断された場合、SySL は従来の著作権ライセンスとして機能します。
2. **契約法**：コードが著作権の対象とならない場合、SySL は拘束力のある契約として機能します。ソフトウェアを使用することにより、あなたは条件に同意します。

この二重根拠アプローチは、SentryのFair Source License (FSL) をモデルとしており、これは複数の企業によって採用され、異議申し立てが成功したことはありません。

### 著作権法が変更された場合はどうなりますか？

裁判所または立法機関がAI生成コードは著作権の対象となると判断した場合、SySL の著作権許諾（第3条）が直ちに効力を生じ、明確なライセンス経路を提供します。ライセンスの修正は必要ありません。

### SySL の下での「著作者」とは誰ですか？

著作者は、誰がAI生成を指示したかに応じて、三つの階層で決定されます：

1. **直接指示**：プロンプトを直接書き、アーキテクチャを設計し、AIを導いた人間。
2. **間接指示**：AIシステム（人間ではなく）が生成を直接指示した場合、そのAIシステムを設計、設定、または展開した人間。
3. **人間の関与なし**：いかなるレベルにおいても人間が関与しなかった場合、ソフトウェアを最初に公表した者。

この多層的アプローチは、将来AIシステム（自身のエージェントフリートなど）が自律的にコード生成を指示する可能性があることを認識しています。この定義により、契約のための法的な拠り所が常に存在することが保証されます。

### なぜ準拠法が日本に設定されているのですか？

SySL の著作者は、日本の法律が契約に基づくソフトウェアライセンスに安定した予測可能な枠組みを提供する法域に拠点を置いています。準拠法として日本を選択したことは意図的なものです：日本はよく発達した契約法体系を持ち、当事者自治を尊重する法制度を有し、SySL が使用される可能性の高い主要法域（米国、EU、中国）に対して中立です。

他の法域の利用者は、現地の消費者保護およびデータ保護の権利を保持します。

### AI開示要件に同意できない場合はどうすればよいですか？

開示要件は SySL の中核です。AI生成を開示することができない、または開示したくない場合は、SySL でライセンスされたソフトウェアを使用すべきではありません。AI開示を要求しない代替ライセンスには、MIT、Apache-2.0、BSDがあります。

## 実践的な質問

### AI_DISCLOSURE ファイルを作成するにはどうすればよいですか？

リポジトリのルートに `AI_DISCLOSURE` または `AI_DISCLOSURE.md` という名前のファイルを作成します。最小限の例：

```
AI MODELS USED:
- Claude Opus 4.6 (Anthropic)
- GLM-4 (Zhipu AI)

TOKENS CONSUMED:
- Approximately 100 billion tokens total

AI GENERATION PROPORTION:
- ~100% of source code is AI-generated
- Architecture design and quality assurance: human
```

### 自分のプロジェクト用に SySL の本文を改変できますか？

ライセンス本文を自身の使用のために改変することはできますが、改変したバージョンを許可なく「合成ソースライセンス (Synthetic Source License)」または「SySL」と呼んではなりません。派生ライセンスを作成する場合は、別個の名称を付けてください。

著作者は SySL の公式改訂版を公表することがあります。ライセンスの第16条を参照してください。
