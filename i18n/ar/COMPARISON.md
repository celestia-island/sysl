## تحليل تفصيلي

### SySL مقابل MIT

**متى تختار MIT**: كودك مكتوب بشريًا، وتريد أقصى درجات البساطة والتساهل، ولا تحتاج إلى حماية براءة الاختراع أو شروط خاصة بالذكاء الاصطناعي.

**متى تختار SySL**: كودك متولد بالذكاء الاصطناعي إلى حد كبير، وتريد أن يعرف المستخدمون ذلك، وتريد أن تعمل الرخصة حتى لو لم تنطبق حقوق الطبع والنشر.

**الفرق الجوهري**: MIT هي رخصة حقوق طبع ونشر بحتة. إذا لم تغطِّ حقوق الطبع والنشر الكود المتولد بالذكاء الاصطناعي الخاص بك، فلا توفّر MIT أي إطار قانوني. تعمل SySL كرخصة حقوق طبع ونشر وكعقد في آن واحد.

### SySL مقابل Apache 2.0

**متى تختار Apache 2.0**: كودك مكتوب بشريًا، وتريد رخصة متساهلة مع حماية براءة اختراع، وتعمل في بيئة مؤسسية.

**متى تختار SySL**: نفس مقارنات MIT، بالإضافة إلى: صُممت SySL لتكون متوافقة مع Apache 2.0. يمكنك دمج الكود المرخص بموجب SySL مع الكود المرخص بموجب Apache-2.0 والتوزيع بموجب Apache 2.0، شريطة الحفاظ على إفصاحات الذكاء الاصطناعي.

**الفرق الجوهري**: تمتلك Apache 2.0 منحًا لبراءة الاختراع. تمتلك SySL منحًا مكافئًا لبراءة الاختراع. الفرق الحاسم هو أن Apache 2.0 تفترض حقوق الطبع والنشر ولا توفّر احتياطيًا تعاقديًا. يعمل منح براءة الاختراع في SySL بشكل مستقل عن حقوق الطبع والنشر.

### SySL مقابل GPLv3

**متى تختار GPLv3**: تريد حقوقًا متروكة قوية — أي شخص يعدّل ويُعيد توزيع كودك يجب أن يوزّع تعديلاته أيضًا بموجب GPL.

**متى تختار SySL**: تريد الشفافية لكن دون إكراه على مشاركة المصدر. تطلب "الحقوق المتروكة التركيبية" في SySL فقط الحفاظ على إفصاح الذكاء الاصطناعي، لا أن يُفتح مصدر المشروع بأكمله.

**الفرق الجوهري**: الحقوق المتروكة في GPL تدور حول إبقاء الكود مفتوحًا. الحقوق المتروكة في SySL تدور حول إبقاء سلسلة إفصاح الذكاء الاصطناعي سليمة. يمكنك أخذ كود SySL، وإضافة تعديلات احتكارية، والتوزيع بموجب رخصة احتكارية — طالما بقي إفصاح الذكاء الاصطناعي.

### SySL مقابل BUSL 1.1

**متى تختار BUSL 1.1**: تريد حماية منتج تجاري من الاستنساخ من قبل المنافسين، مع تحول الكود في النهاية إلى مصدر مفتوح.

**متى تختار SySL**: تريد أقصى درجات التبني مع معيار شفافية. لا توجد فترة عدم منافسة ولا تحويل تلقائي.

**الفرق الجوهري**: BUSL رخصة دفاعية (حماية العمل التجاري). SySL رخصة شفافية (إرساء معيار اجتماعي). تخدم كل منهما غرضًا مختلفًا ويمكن أن تتعايشا عبر الترخيص المزدوج (SySL §11.3). بالنسبة للمشاريع المتولدة بالذكاء الاصطناعي التي تستخدم كليهما، ينبغي تعيين رخصة التحويل في BUSL إلى SySL — لا Apache-2.0 — وإلا فإن معيار الشفافية يتلاشى عند انتهاء فترة عدم المنافسة.

## مصفوفة التوافق

هل يمكنك دمج كود بموجب الرخصة A مع كود بموجب الرخصة B؟

| الرخصة A ↓ / الرخصة B → | MIT | Apache 2.0 | GPLv3 | SySL 1.0 | BUSL 1.1 |
|:---|:---:|:---:|:---:|:---:|:---:|
| **MIT** | ✓ | ✓ | ✓ | ✓ | ✓ |
| **Apache 2.0** | ✓ | ✓ | ✓ (الإصدار 3 فقط) | ✓ | مشروطة |
| **GPLv3** | ✓ | ✓ (الإصدار 3 فقط) | ✓ | قيد المراجعة | ✗ |
| **SySL 1.0** | ✓ | ✓ | قيد المراجعة | ✓ | مشروطة |
| **BUSL 1.1** | ✓ | مشروطة | ✗ | مشروطة | ✗ |

*✓ = توافق معروف. ✗ = عدم توافق معروف. "قيد المراجعة" = لم يُحلَّل بعد. "مشروطة" = يعتمد على شروط محددة.*

## أي رخصة يجب أن أستخدم؟

```
هل كودك متولد بالذكاء الاصطناعي إلى حد كبير؟
  ├── نعم → هل تطمئن إلى ألا يعرف المستخدمون؟
  │         ├── نعم → MIT أو Apache 2.0
  │         └── لا  → SySL
  └── لا  → هل تحتاج إلى حماية براءة اختراع؟
              ├── نعم → Apache 2.0
              └── لا  → MIT
```

للكود المتولد بالذكاء الاصطناعي تحديدًا:

```
هل تريد أن يحافظ العمل المشتق على إفصاح الذكاء الاصطناعي؟
  ├── نعم → SySL
  └── لا  → MIT أو Apache 2.0 (مع الإقرار بعدم اليقين القانوني)
```
