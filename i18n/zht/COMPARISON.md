## 詳細分析

### SySL 與 MIT 對比

**何時選擇 MIT**：您的程式碼由人類編寫，您希望最大程度的簡潔與寬容，且您不需要專利保護或 AI 特定條款。

**何時選擇 SySL**：您的程式碼主要由 AI 生成，您希望使用者了解這一點，且您希望授權條款即使在著作權不適用時也能發揮作用。

**關鍵區別**：MIT 純粹是著作權授權條款。若著作權不涵蓋您 AI 生成的程式碼，MIT 無法提供法律框架。SySL 則同時作為著作權授權條款與合約運作。

### SySL 與 Apache 2.0 對比

**何時選擇 Apache 2.0**：您的程式碼由人類編寫，您希望獲得帶有專利保護的寬容授權條款，且您處於企業環境中。

**何時選擇 SySL**：與 MIT 對比的情形相同，此外：SySL 被設計為與 Apache 2.0 相容。您可以將 SySL 授權的程式碼與 Apache-2.0 授權的程式碼結合，並以 Apache 2.0 散布，前提是保留 AI 生成披露。

**關鍵區別**：Apache 2.0 提供專利授權。SySL 提供等效的專利授權。關鍵區別在於，Apache 2.0 假定著作權存在且不提供合約後備。SySL 的專利授權獨立於著作權運作。

### SySL 與 GPLv3 對比

**何時選擇 GPLv3**：您希望強著佐權——任何修改並重新散布您程式碼的人都必須也以 GPL 散布其修改。

**何時選擇 SySL**：您希望透明度，但不希望強制共享原始碼。SySL 的「合成式著佐權」僅要求保留 AI 生成披露，而非將整個專案開源。

**關鍵區別**：GPL 的著佐權旨在保持程式碼開放。SySL 的著佐權旨在保持 AI 生成披露鏈完整。您可以取得 SySL 程式碼，加入專有修改，並以專有授權條款散布——只要 AI 生成披露得以保留。

### SySL 與 BUSL 1.1 對比

**何時選擇 BUSL 1.1**：您希望保護商業產品免受競爭對手複製，並最終將程式碼轉換為開源。

**何時選擇 SySL**：您希望在透明度規範下獲得最大採用率。沒有競業禁止期間，也沒有自動轉換。

**關鍵區別**：BUSL 是防禦性授權條款（保護業務）。SySL 是透明度授權條款（建立社會規範）。它們服務於不同目的，可以透過雙重授權共存（SySL §11.3）。對於同時使用兩者的 AI 生成專案，BUSL 的變更授權條款應設為 SySL——而非 Apache-2.0——否則透明度規範將在競業禁止期間結束後消失。

## 相容性矩陣

您可以將授權條款 A 下的程式碼與授權條款 B 下的程式碼結合嗎？

| 授權條款 A ↓ / 授權條款 B → | MIT | Apache 2.0 | GPLv3 | SySL 1.0 | BUSL 1.1 |
|:---|:---:|:---:|:---:|:---:|:---:|
| **MIT** | ✓ | ✓ | ✓ | ✓ | ✓ |
| **Apache 2.0** | ✓ | ✓ | ✓（僅 v3） | ✓ | 有條件 |
| **GPLv3** | ✓ | ✓（僅 v3） | ✓ | 審核中 | ✗ |
| **SySL 1.0** | ✓ | ✓ | 審核中 | ✓ | 有條件 |
| **BUSL 1.1** | ✓ | 有條件 | ✗ | 有條件 | ✗ |

*✓ = 已知相容。✗ = 已知不相容。「審核中」= 尚未分析。「有條件」= 取決於具體條款。*

## 我應該使用哪個授權條款？

```
Is your code substantially AI-generated?
  ├── Yes → Are you comfortable with users not knowing?
  │         ├── Yes → MIT or Apache 2.0
  │         └── No  → SySL
  └── No  → Do you need patent protection?
              ├── Yes → Apache 2.0
              └── No  → MIT
```

針對 AI 生成程式碼：

```
Do you want derivatives to keep AI disclosure?
  ├── Yes → SySL
  └── No  → MIT or Apache 2.0 (acknowledging legal uncertainty)
```
